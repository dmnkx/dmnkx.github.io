doobeom

업무 자동화 인프라

Docker Compose 단일 스택으로 프록시·워크플로 오케스트레이션·브라우저 자동화·FastAPI·관측성을 internal_net 하나에 묶은 Dev 환경입니다.

Stack Docker Compose · Bridge
Scope dev.doobeom.io
Services 11 containers
Source docker-compose.yml

1. 개요

단일 호스트에서 운영되는 자동화 플랫폼의 역할 분리와 계층 구조입니다.

FIG 1-1 · 5계층 역할 분리

본 인프라는 사내/업무용 자동화를 위한 단일 호스트 Docker Compose 구성입니다. 모든 서비스는 internal_net (bridge)에 연결되어 컨테이너 이름(DNS)으로 상호 호출하며, 외부 노출은 Nginx Proxy Manager(NPM)와 호스트 포트 매핑을 통해 이뤄집니다.

  • Edge — Nginx Proxy Manager: TLS 종료, 리버스 프록시, Proxy Host 관리 UI
  • Automation Core — n8n 워크플로 엔진 + external task runners + Playwright worker + FastAPI
  • Domain APIs — automation-api (GHCR 이미지, PG 정산·Google Spread/SheetTab)
  • Adjacent — legal 전용 n8n, Appsmith EE (데이터·역할 격리)
  • Observability — cAdvisor → Prometheus → Grafana 파이프라인

2. 시스템 구성도

Internet에서 NPM을 거쳐 internal_net 상의 서비스로 트래픽이 분기됩니다. 점선은 브로커·내부 API·메트릭 스크레이프 경로입니다.

FIG 2-1 · 전체 토폴로지 (11 services)

Edge Automation Browser Legal Appsmith Observability

컨테이너 간 호출 시에는 호스트 포트가 아닌 컨테이너 포트를 사용합니다. 예: n8n → http://automation-api:8000, n8n → http://playwright-worker:3101. 호스트에 매핑된 8001, 5678 등은 외부 접속·디버그용입니다.

3. 도메인 · 진입점

compose에 명시된 호스트명과 NPM Proxy Host 설정. TLS는 Edge에서 종료합니다.

5. Docker 이미지

각 서비스가 사용하는 이미지의 출처, 커스텀 여부, 역할과 선택 이유를 정리합니다. 커스텀 3종(n8n, runners, playwright-worker)과 GHCR 배포 API를 중점적으로 다룹니다.

FIG 5-1 · 이미지 공급 경로 (Official / Build / GHCR)

nginx jc21/nginx-proxy-manager:latest

type: official registry · service: nginx

Nginx Proxy Manager 공식 이미지입니다. Nginx + Let's Encrypt + 웹 UI를 한 컨테이너로 제공해, 도메인별 Proxy Host·SSL 발급을 UI로 관리합니다. 호스트 :80/:443으로 트래픽을 받고, 관리 UI는 :8000→81로 노출합니다. 데이터는 ./data/nginx/data, 인증서는 ./data/nginx/letsencrypt에 영속화합니다.

  • 선택 이유: TLS·리버스 프록시를 코드 없이 설정, Kong 대비 운영 부담 적음
  • 내부 네트워크에서 컨테이너 이름·포트로 upstream 지정

appsmith appsmith/appsmith-ee:v1.99

type: official (EE pinned) · service: appsmith

Appsmith Enterprise 고정 버전 이미지입니다. 로우코드로 내부 어드민·업무 화면을 구성합니다. 스택 데이터는 /data/appsmith를 /appsmith-stacks에 마운트해 유지합니다. 호스트 :8888→80으로 접근합니다.

  • 버전 pin(v1.99)으로 업그레이드 시 회귀를 통제
  • 자동화 스택과 같은 internal_net에 두어 n8n·API와 연동 가능

n8n build ./images/n8n ← n8nio/n8n:latest

type: custom Dockerfile · service: n8n

공식 n8nio/n8n 위에 playwright-core와 axios를 전역 설치한 커스텀 이미지입니다. Code node에서 외부 npm 패키지를 쓰려면 NODE_FUNCTION_ALLOW_EXTERNAL과 NODE_PATH가 필요해, 베이스 이미지만으로는 부족합니다. External runners·trust proxy·네이티브 Python runner도 compose env로 활성화합니다.

  • 베이스: n8nio/n8n:latest → root로 npm i -g playwright-core axios → USER node
  • broker :5679로 n8n-runners와 통신 · 데이터 /data/n8n
  • 파일 공유: /data/shared-downloads · shm_size: 2gb

n8n-runners build ./images/runners ← n8nio/runners:latest

type: custom Dockerfile · service: n8n-runners

n8n External Task Runner 공식 이미지를 확장합니다. JS runner에는 moment·uuid·playwright·axios와 Chromium을, Python runner에는 numpy·pandas·httpx를 넣어 Code/파이썬 워크플로의 무거운 작업을 메인 n8n 프로세스와 분리합니다. n8n-task-runners.json으로 runner 설정을 주입합니다.

  • N8N_RUNNERS_TASK_BROKER_URI=http://n8n:5679 · auth token으로 broker 연결
  • 포트 5680/5681(이미지 EXPOSE) · depends_on: n8n · shm 2gb
  • 메인 n8n을 가볍게 유지하고 runner에서 브라우저·데이터 처리 실행

playwright-worker build ./images/playwright-worker ← mcr.microsoft.com/playwright:v1.59.1-jammy

type: custom Dockerfile · service: playwright-worker

Microsoft 공식 Playwright 이미지(브라우저 바이너리 포함) 위에 Node 앱을 올립니다. n8n과 별도 HTTP API(:3101)로 브라우저 시나리오를 실행하고, 스크립트는 /data/browserless-scripts, 임시 파일은 /data/playwright, 산출물은 shared-downloads를 사용합니다. n8n과는 N8N_BASE_URL로 콜백합니다.

  • 베이스: Playwright v1.59.1 Jammy · npm install 후 npm start
  • n8n Code node의 인라인 브라우저와 역할 분리 — 장시간·재사용 시나리오에 적합
  • shm_size: 2gb로 Chromium 안정성 확보

legal docker.n8n.io/n8nio/n8n:latest

type: official registry · service: legal

메인 n8n과 분리된 두 번째 n8n 인스턴스입니다. 커스텀 패키지 없이 공식 이미지를 그대로 쓰며, 호스트·웹훅을 legal.dev.doobeom.io / 포트 6678에 맞춥니다. 데이터는 /data/legal만 사용해 메인 워크플로·자격증명과 공간을 완전히 나눕니다.

  • 선택 이유: 민감 도메인(법무 등)을 동일 호스트에서도 격리 운영
  • 메인처럼 runners/playwright를 붙이지 않는 최소 구성

automation-api ghcr.io/doobeom-platform/automation-api:latest

type: CI → GHCR (amd64/arm64) · base: python:3.14 · service: automation-api

자체 개발 FastAPI 서비스입니다. Dockerfile은 python:3.14에 pip로 dependencies를 설치하고 non-root app 유저로 uvicorn을 띄웁니다. GitHub Actions가 이미지를 빌드해 GHCR에 푸시하며, compose는 :latest를 pull합니다. PG 정산 조회, Google Drive 스프레드시트/시트 탭 ensure API를 제공합니다.

  • 포트 8001:8000 · 내부 호출은 http://automation-api:8000
  • 인증: Google SA JSON 마운트 또는 env · Clean Architecture 구조의 Python 앱
  • 로컬/서버 모두 multi-arch 이미지로 Apple Silicon·linux/amd64 지원

cadvisor gcr.io/cadvisor/cadvisor:latest

type: official · service: cadvisor

Google cAdvisor 이미지로 컨테이너·호스트 리소스 메트릭을 export합니다. rootfs, docker sock, sys 등을 read-only로 마운트해 런타임 상태를 수집하고, Prometheus가 cadvisor:8080을 scrape합니다.

prometheus prom/prometheus:latest

type: official · service: prometheus

메트릭 TSDB입니다. 호스트의 ./prometheus.yml을 마운트해 job docker-containers → cadvisor:8080, scrape interval 5s(개발용; 운영은 15s 권장)으로 수집합니다. UI/쿼리는 호스트 :9090.

grafana grafana/grafana:latest

type: official · service: grafana

대시보드·시각화 UI입니다. Prometheus를 datasource로 붙여 컨테이너 CPU/메모리 등을 모니터링합니다. 대시보드 설정은 named volume grafana-storage에 보존하며, 포트 :3000으로 접근합니다.

6. 주요 실행 흐름

실무에서 자주 타는 경로. 각 흐름마다 도식과 단계 설명을 함께 둡니다.

A. Appsmith → n8n 어드민 상품 등록

  1. 사내 직원이 Appsmith UI(:8888)에 로그인 — n8n UI를 직접 쓰지 않음
  2. Appsmith 앱(폼·버튼)이 internal_net 경유 또는 NPM 도메인으로 n8n 웹훅/워크플로를 호출
  3. n8n이 상품 등록용 자동화 워크플로 실행 (검증·변환·외부 어드민 API)
  4. 필요 시 n8n이 playwright-worker로 브라우저 UI 조작, automation-api(FastAPI)로 정산/시트 연동을 호출
  5. 어드민에 상품이 반영되고, 결과를 Appsmith에 성공/실패로 표시

핵심: Appsmith는 업무 화면, n8n은 오케스트레이터, playwright-worker·FastAPI는 n8n이 위임하는 전문 워커입니다.

B. 워크플로 + External Runner

  1. 사용자가 n8n.dev.doobeom.io 또는 웹훅으로 워크플로 실행
  2. n8n이 Task Broker :5679로 n8n-runners에 작업 위임
  3. Runner가 JS(Chromium 포함) / Python 태스크 수행 (N8N_RUNNERS_AUTH_TOKEN 인증)
  4. 산출 파일은 /data/shared-downloads로 n8n·playwright와 공유

C. playwright-worker 동작

  1. 호출자(주로 n8n, 또는 Appsmith 경유 워크플로)가 http://playwright-worker:3101로 HTTP 요청
  2. 워커가 SCRIPTS_DIR(/data/browserless-scripts)에서 지정 스크립트를 찾아 실행
  3. 내장 Chromium(Playwright v1.59)으로 로그인·클릭·입력·다운로드 등 UI 자동화 수행
  4. 산출물(파일·JSON)을 shared-downloads 또는 응답 body로 반환 · 필요 시 N8N_BASE_URL로 n8n에 콜백
  5. n8n Code node 인라인 브라우저와 분리 — 장시간·재사용 시나리오와 shm 2gb가 필요한 작업에 사용

어드민 상품 등록처럼 “공식 API가 없거나 UI로만 가능한 작업”은 playwright-worker가 담당하고, n8n은 타임아웃·재시도·후속 단계를 오케스트레이션합니다.

D. automation-api (FastAPI) 동작

  1. 호출자(주로 n8n HTTP Request 노드)가 http://automation-api:8000으로 REST 호출 (호스트 디버그는 :8001)
  2. PG — GET /pg/naverpay?date=YYYYMMDD로 건별 정산 조회, 가공(컬럼명·HTML unescape 등) 후 JSON 반환
  3. Spread — PUT /google/ensure/spread/{date}로 PG_{date} 스프레드시트 id를 조회·없으면 생성
  4. Sheet tab — PUT /google/ensure/sheettab/{spreadId} + body {"name":"..."}으로 하단 탭 id 확보
  5. Google 호출은 서비스 계정 자격증명으로 수행 · Google 통신 성공 시 HTTP 200 + body success로 결과 구분

FastAPI는 n8n Code 노드에 자격증명·비즈니스 로직을 넣지 않고, 재사용 가능한 HTTP API로 분리한 레이어입니다. Playwright가 “브라우저로만 되는 일”을 맡는 반면, FastAPI는 “공식 API·시트 id” 같은 구조화된 연동을 맡습니다.

E. 관측 파이프라인

  1. cAdvisor가 컨테이너 메트릭 expose (:8080)
  2. Prometheus가 prometheus.yml 기준 5초 간격 scrape
  3. Grafana(:3000)에서 Prometheus datasource 시각화

8. 운영 · 보안 · 확장

FIG 8-1 · 운영 설계 원칙

  • Edge에서 TLS 종료 후 내부는 HTTP·서비스명 통신 — 단일 호스트/VPC 패턴
  • n8n External Runner: broker 인증 토큰 (N8N_RUNNERS_AUTH_TOKEN)
  • legal 인스턴스 분리 — 민감 워크플로와 일반 자동화 데이터 공간 분리
  • 모니터링: cAdvisor 기반, scrape 5s (개발 환경용, 운영 시 15s 권장)
  • API Gateway(Kong) 없이 NPM 경로/호스트 라우팅 — 인증·쿼터 필요 시 이후 검토

확장 방향

  • api.dev.doobeom.io 단일 엔트리 + path 라우팅 (/pg, /google → automation-api)
  • automation-api: 서비스 계정 JSON 마운트 또는 GOOGLE_APPLICATION_CREDENTIALS
  • 이미지 갱신: CI push 후 서버 docker compose pull && up 또는 Watchtower 검토
