2. 전체 아키텍처

서비스 workflow는 얇게 유지하고, 배포·알림·설정 조회는 doobeom-sys/actions로 추상화합니다. 환경별 설정은 SSM /doobeom/{repo}/env/{env}에서 읽습니다.

5. PHP

커머스·어드민·미디어 웹을 Composer로 패키징한 뒤 CodeDeploy로 EC2에 배포합니다. 호스트 단위(STG 슬롯)로 workflow를 분리해 동시 배포 충돌을 피합니다.

패턴: SSM → appspec/hooks 치환 → Composer install → ZIP → S3 → CodeDeploy + Tag + Slack
커머스 웹 (Storefront) commerce/db-storefront-web · prod/QA/STG www1~4 · php-fpm 재시작 옵션
어드민 웹 commerce/db-admin-web · admin1~4.stg · 동일 CodeDeploy 계약
미디어 웹 media/db-media-web · create-appspec · manual redeploy

6. Java

Gradle 기반 REST API·배치 워커·공유 라이브러리, Android 앱 빌드에 Actions를 사용합니다. API는 CodeDeploy, Batch는 ECR 이미지와 Step Functions 실행을 분리합니다.

REST API SSM → Gradle build → upload-s3 → CodeDeploy → update-version → Slack
API 서비스 db-admin-api · db-fulfillment-api · db-library-api · tokyo 전용 포함
Batch 이미지 publish와 submit-job 분리 · workflow_call 로그 Job
Android profile 선택 → OIDC/Secrets → Gradle + Fastlane

7. Python

같은 Python이라도 서버 프로세스와 Lambda는 배포 대상·Job 구조·롤백 방식이 다릅니다.

일반 vs Lambda 비교

공통: SSM → S3 → CodeDeploy → Slack. 차이: 서버 파일 교체 vs Alias 트래픽 전환, 2 Job vs 3 Job.
구분일반 Python (서버)Lambda Python
실행 환경EC2 위 Python 앱 / ETLLambda + Alias
Job 구조BuildAndUpload → DeployBuildAndUpload → PublishLambda → Deploy
아티팩트앱 ZIP + appspec함수 ZIP + Layer ZIP
CodeDeploybundleType ZipbundleType YAML
트래픽서버 교체 즉시Job3에서만 Alias 전환
롤백이전 ZIP 재배포Alias를 이전 버전으로
대표db-etl-flow · db-search · db-recommenderdb-barcode-lambda · db-presignup-lambda

② Lambda Python — 3 Job Alias

Job주요 Step운영 영향

Deploy create-appspec → CodeDeploy YAML → Alias 전환 · Slack 3/3 트래픽 전환

9. 하이라이트

재사용 Action 설계 PHP·Java·Python이 동일 배포 계약을 쓰도록 Composite/JS 라이브러리 운영
안전한 서버리스 Lambda publish와 Alias 전환을 분리한 3-Job 블루그린
배포와 실행 분리 Batch 이미지 푸시와 Step Functions 실행을 나누고 Slack·태그로 가시성 확보
