E2E Automation · Playwright 1.49

두범마켓 이커머스
엔드투엔드 자동화 테스트

Microsoft Playwright로 실제 사용자 여정(신상품 확인 → 로그인 → 장바구니 → 주문 → 결제 수단 선택)을 브라우저에서 재현·검증하는 E2E 테스트 스위트입니다. Chromium/WebKit 크로스 브라우저와 iframe·동적 팝업 등 이커머스 UI 과제를 다룹니다.

회사 두범 (doobeom) 대상 doobeommarket.com 프레임워크 @playwright/test 브라우저 Chrome · Safari(WebKit)
본 문서는 포트폴리오용입니다. 회사명·도메인 등 식별 정보는 두범(doobeom)으로 가명 처리되었습니다.

2. 테스트 목표

수동 회귀로 반복하기 비용이 큰 핵심 커머스 플로우를 자동화해, UI/필터/결제 위젯 변경 시 조기 결함을 검출합니다.

배포 파이프라인에서의 역할
flowchart LR
  A[배포 / UI 변경] --> B[Playwright 스위트]
  B --> C{결과}
  C -->|Pass| D[리그레션 안전]
  C -->|Fail| E[스크린샷 · 비디오 · HTML Report]
  E --> F[원인 분석 · 수정]
카탈로그 검증신상품 로드 · 품절 오버레이 · 두범배송 필터 재집계
주문 플로우로그인 → 장바구니 → 배송 타입 → 결제 iframe
UI 장애물 제거시스템 팝업·바텀시트·툴팁 DOM 제거
크로스 브라우저Chrome 채널 + Safari(WebKit) 프로젝트

3. 기술 스택

Playwright는 CDP 및 자체 프로토콜로 브라우저를 제어합니다. Selenium식 WebDriver 대비 빠른 실행·강력한 셀렉터·내장 Auto-wait가 특징입니다.

패키지버전역할
@playwright/test^1.49.0러너, fixture, expect, 프로젝트 설정
playwright^1.49.0브라우저 자동화 코어
dotenv^16.4.7.env에서 USER_ID / USER_PASSWORD 로드
기술 계층
flowchart TB
  L1["Test Specs · Assertion · dotenv"]
  L2["playwright/test Runner · Fixture · HTML Reporter"]
  L3["Playwright Core · Locator · Frame · evaluate"]
  L4["Browser · Chromium/Chrome · WebKit"]
  L5["Target · doobeommarket.com · 결제 iframe"]
  L1 --> L2 --> L3 --> L4 --> L5

Test Runner가 Worker에 테스트를 분배하고, 각 Worker는 Browser → Context → Page 순으로 격리 환경을 만듭니다. 실패 시 비디오·스크린샷이 HTML 리포트에 첨부됩니다.

4.1 Browser / Context / Page

BrowserContext는 쿠키·스토리지·권한이 격리된 “시크릿 창”에 가깝고, 테스트마다 깨끗한 상태로 시작할 수 있습니다. 시그니처의 ({ page })는 fixture로, Runner가 Context/Page를 주입합니다.

계층 관계
flowchart LR
  B[Browser] --> C1[Context A]
  B --> C2[Context B]
  C1 --> P1[Page]
  C1 --> P2[Page]
  C2 --> P3[Page]

4.2 Auto-wait

Locator 액션(click, fill)은 요소가 attached → visible → stable → enabled → receives events가 될 때까지 대기합니다. 신상품 테스트에서는 waitForSelector(..., { state: 'attached' })로 리스트 마운트를 보완합니다.

Auto-wait 상태 전이
stateDiagram-v2
  [*] --> Attached: DOM에 존재
  Attached --> Visible: display/opacity OK
  Visible --> Stable: 애니메이션 안정화
  Stable --> Enabled: disabled 아님
  Enabled --> Actionable: 포인터 이벤트 수신
  Actionable --> [*]: click / fill

6. 설정 상세

공통 use 옵션과 프로젝트별 브라우저 프로필로 나뉩니다. baseURL을 두면 page.goto('/main/new')처럼 상대 경로로 이동합니다. 

module.exports = {
  projects: [ /* chrome, safari */ ],
  timeout: 30000,
  expect: { timeout: 5000 },
  reporter: [['html', { outputFolder: 'test-results', open: 'never' }]],
  use: {
    baseURL: 'https://doobeommarket.com/',
    viewport: { width: 1280, height: 720 },
    ignoreHTTPSErrors: true,
    video: 'retain-on-failure',
    actionTimeout: 10000,
  },
};

7. 신상품 갯수 체크

파일: tests/countNewProducts.spec.js — 신상품 카드를 순회하며 전체·품절 수량을 집계하고, 두범배송 필터 후 동일 로직으로 재검증합니다.

집계 알고리즘
flowchart TD
  A[goto /main/new] --> B[waitForSelector product items]
  B --> C[Locator.count → N]
  C --> D{i < N}
  D -->|yes| E[nth i]
  E --> F{.sold-out-cover visible?}
  F -->|yes| G[soldOutCount++]
  F -->|no| H[건너뜀]
  G --> D
  H --> D
  D -->|no| I[console.log 전체/품절]
  I --> J[두범배송 필터 클릭]
  J --> B2[getNewProductCount 재호출]
  • Locator 재사용 — 필터 클릭 후에도 같은 셀렉터로 최신 DOM을 읽습니다.
  • nth(i) 순회 — 카드별 품절 오버레이를 인덱스 단위로 검사합니다.
  • 내부 헬퍼 — getNewProductCount로 필터 전후 DRY 재사용.
const isSoldOut = await product
  .locator('.sold-out-cover')
  .isVisible();
// 필터: label[for="doobeom_delivery_check"]

8. 로그인 · 주문 · 결제

파일: tests/loginAndOrderPayMethod.test.js — 인증 Happy Path를 끝까지 밟아 주문 페이지와 결제 위젯 도달을 검증합니다.

시퀀스
sequenceDiagram
  participant T as Test
  participant P as Page
  participant S as doobeommarket.com
  participant I as 결제 iframe

  T->>P: goto /ko/login
  T->>P: fill user_id / user_password
  T->>P: click submit
  P->>S: POST 로그인
  S-->>P: redirect /ko/main
  T->>P: goto /ko/cart
  T->>T: expect(productCount).toBeGreaterThan(0)
  T->>P: deleteSystemPopup()
  T->>P: 일반배송 선택 → 주문하기
  P->>S: navigate /ko/order
  T->>P: deleteSystemPopup()
  T->>I: contentFrame()
  T->>I: 일반결제 → 가상계좌 클릭
  • 인증 — waitForURL('/ko/main')으로 리다이렉트 결과를 동기화
  • 장바구니 가드 — 빈 카트면 Fail-fast
  • 텍스트 Locator — label.filter({ hasText: '일반배송' })
사전 조건: 계정 장바구니에 상품이 이미 담겨 있다고 가정합니다.

9. iframe 제어 — 결제 위젯

결제 UI는 별도 origin iframe에 렌더됩니다. Frame 객체를 얻은 뒤 그 위에서 Locator를 사용합니다.

부모 Document ↔ 결제 iframe
flowchart TB
  subgraph Parent["Parent · /ko/order"]
    PM["#payment-method"]
    IF["iframe"]
  end
  subgraph Frame["iframe ContentDocument"]
    L1["label: 일반결제"]
    B1["button: 가상계좌"]
  end
  PM --> IF
  IF -.->|contentFrame| Frame
  L1 --> B1
          
await page.waitForSelector('#payment-method');
const iframe = await page.locator('#payment-method iframe').contentFrame();
await iframe.locator('label').filter({ hasText: '일반결제' }).click();
await iframe.locator('button').filter({ hasText: '가상계좌' }).click();

11. 환경 변수 · 자격증명

USER_ID, USER_PASSWORD를 .env에서 읽습니다. 리포지토리에 비밀을 커밋하지 않습니다. 

require('dotenv').config();
const user = {
  id: process.env.USER_ID,
  password: process.env.USER_PASSWORD,
};
.env는 .gitignore에 포함하고, HTML 리포트·트레이스에 비밀번호가 남지 않도록 합니다.
